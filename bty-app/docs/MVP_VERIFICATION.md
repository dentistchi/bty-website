# MVP 진행 · 검증 전략

변경 사항을 전부 한 번에 확인하기 어렵고, 오래 써봐야 드러나는 것들이 있다.  
**MVP 진행하면서 체크해도 된다.** 나중에 발견해도 대부분 수정 가능하다.

---

## 1) 지금 해두면 좋은 최소 검증 (한 번만)

배포 직후 **한 번만** 아래만 확인해 두면, 나중에 “코드가 꼬였나?” 할 때 기준이 된다.

**프로덕션 URL**  
- **https://bty-website.ywamer2022.workers.dev**  
- 한국어: `https://bty-website.ywamer2022.workers.dev/`  
- 영어: `https://bty-website.ywamer2022.workers.dev/en`  

(Worker 이름: `bty-website`. 계정 서브도메인이 바뀌면 위 URL의 `ywamer2022` 부분만 해당 계정 값으로 바꾸면 됨.)

| 항목 | 방법 |
|------|------|
| 빌드/배포 성공 | GitHub Actions에서 `Deploy to Cloudflare Workers` 녹색 확인 |
| 로그인 가능 | 프로덕션(아래 URL)에서 로그인 → `/en/bty/(protected)/dashboard` 또는 `/en/bty-arena` 진입 |
| Arena 1회 플로우 | 시나리오 하나 골라서 선택 → Reflection → 완료까지 한 번 진행 |

이 세 가지가 되면 “배포·인증·Arena 루프”는 동작하는 상태로 고정해 두는 것이다.

---

## 2) MVP 사용하면서 점검해도 되는 것들

- Tier 25/50/75 모달, 서브네임 변경 (상위 5%일 때만 등)
- 주간 성찰 퀘스트, 개인 기록, 시즌 10% 캐리오버
- 윤리적 선택 +5 XP, Tier 75 전용 시나리오
- 시나리오 인트로에서 메타 문구(hidden risk 등) 안 보이는지

→ **실제로 쓰다가 “이거 안 되네?” 하면 그때 이슈로 남기고 고치면 된다.**  
지금 전부 시나리오를 돌려보지 않아도 된다.

---

## 3) 나중에 고치기 어려운가? 코드가 꼬일까?

- **대부분은 나중에 고쳐도 된다.**  
  지금 넣은 기능은 “플래그/조건 하나 더 넣은 것”이나 “API 하나 추가” 수준이라, 나중에 버그/요구사항 바뀌어도 수정 범위가 명확하다.
- **진짜 꼬이기 쉬운 부분만 조심하면 된다.**  
  - **Supabase Auth / 쿠키:** 로그인·세션·쿠키 규격을 함부로 바꾸면 “어디서부터 안 되는지” 찾기 어려울 수 있다.  
    → 그래서 `CONTEXT.md`에 “한 곳에서만 쿠키 set/clear” 같은 원칙을 적어 둔 것이다.  
  - **DB 스키마:** 마이그레이션 없이 컬럼 이름/타입을 바꾸면 런타임 에러가 난다.  
    → 스키마 변경은 항상 마이그레이션 파일 추가 + 배포 전에 로컬/스테이징에서 한 번 실행해 보는 게 안전하다.

정리하면, **지금 “한 번만” 최소 검증만 해 두고, 나머지는 MVP 진행하면서 체크해 보라고 하면 된다.**  
그때 적용 안 된 걸 발견해도, 위 두 가지(인증/쿠키, DB 스키마)만 지키면 고치기가 크게 어렵지 않다.

---

## 4) 다음에 진행할 것 (우선순위 제안)

1. **배포 확인**  
   - `main`에 push → Actions 성공 → 프로덕션 URL에서 로그인 + Arena 1회 플로우 (위 1) 최소 검증).
2. **한국어 버전**  
   - 뼈대가 잡혀 있으니, 문자열/로어를 ko로 채우는 작업.
3. **MVP 사용 피드백**  
   - 실제 사용하면서 “이거 안 됐다 / 이거 바꾸면 좋겠다” 리스트만 모으고, 그걸로 이슈/작업 정리.

이 순서로 진행하면 된다.
