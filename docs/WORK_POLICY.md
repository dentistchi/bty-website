# BTY 작업 정책 — 일상 vs 배포 시 검증

**목적:** 매일 하는 작업과 배포 시 검증을 구분해, 프로젝트 완성(기능 작업)에 집중할 수 있게 한다.

---

## 1. 일상 작업 (매일)

- **내용:** 프로젝트를 **만들고 완성**하는 작업.
- **우선순위:** 웹 개발 — **UI·API·도메인** (기능 구현, 버그 수정, Q3·Q4 로드맵 작업).
- **예시:** LE Stage, AIR API, 대시보드, Elite 멘토 승인/거절, Healing/Awakening, 로드맵·백로그 정리 등.
- **참고 문서:** `docs/ROADMAP_Q3_Q4.md` — 여기 있는 기능 스프린트를 일상에서 진행.

**매 스프린트마다** Release Gate + 문서 점검 + 접근성 + 테스트를 반복할 **필요는 없다**.  
정책상 그건 **배포 전 1회**가 원칙이다.

---

## 2. 배포 시 검증 (배포할 때)

- **시점:** 실제로 **배포를 하기로 했을 때** 1회 수행.
- **내용:**  
  - Release Gate (bty-release-gate.mdc A~F)  
  - 문서·백로그 점검 (BTY_RELEASE_GATE_CHECK, 보드, CURRENT_TASK 반영)  
  - 린트·테스트·빌드·(선택) 접근성·배포 환경 확인  
- **체크리스트:** `docs/MVP_DEPLOYMENT_READINESS.md`  
- **결과 기록:** `docs/BTY_RELEASE_GATE_CHECK.md`에 “배포 전 1회” 점검 1건 기록.

**배포할 때마다 검증하는 과정**은, 필요에 따라 나중에 도입할 수 있다.  
현재는 **일상 = 기능 작업**, **검증 = 배포 전 1회**로 둔다.

---

## 3. 정리

| 구분 | 내용 |
|------|------|
| **일상** | UI·API·도메인 작업, Q3·Q4 기능 스프린트. 프로젝트 완성에 집중. |
| **배포 시** | MVP 배포 준비 체크리스트 1회 수행, Gate·문서 반영. |
| **매 스프린트** | Gate + 문서 + 접근성 + 테스트 반복은 정책상 필수 아님. |
